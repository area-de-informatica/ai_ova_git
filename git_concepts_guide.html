Branching (Ramificación)

Trabajando con múltiples líneas de desarrollo

¿Qué es el Branching?

Las ramas (branches) en Git te permiten divergir de la línea principal de desarrollo y trabajar en características, experimentos o correcciones sin afectar el código principal.

Conceptos Clave:
  • main/master: La rama principal del proyecto
  • feature branch: Rama para desarrollar nuevas características
  • hotfix branch: Rama para correcciones urgentes
  • develop branch: Rama de desarrollo activo
Comandos Básicos:
git branch

Lista todas las ramas locales

git branch nueva-caracteristica

Crea una nueva rama llamada "nueva-caracteristica"

git checkout -b nueva-caracteristica

Crea y cambia a una nueva rama en un solo comando

git checkout main

Cambia a la rama main

* main ├── commit 1: Initial commit ├── commit 2: Add basic structure └── commit 3: Setup configuration * feature/nueva-caracteristica ├── commit 4: Start new feature └── commit 5: Implement feature logic
Tip: Usa nombres descriptivos para tus ramas como "feature/login-system" o "hotfix/security-patch"

Merging (Fusión)

Integrando cambios de diferentes ramas

¿Qué es el Merging?

El merge es el proceso de integrar cambios de una rama hacia otra. Es fundamental para combinar el trabajo realizado en diferentes características.

Tipos de Merge:

Ocurre cuando la rama de destino no ha tenido commits adicionales desde que se creó la rama de origen.

main: A---B---C \ feature: D---E

Después del merge:

main: A---B---C---D---E

Ocurre cuando ambas ramas han divergido y tienen commits únicos.

main: A---B---C---F \ \ feature: D---E---M (merge commit)
Comandos de Merge:
git checkout main
git merge feature/nueva-caracteristica

Fusiona la rama "feature/nueva-caracteristica" en la rama actual (main)

Importante: Siempre asegúrate de estar en la rama de destino antes de hacer el merge.

Stashing (Almacenamiento Temporal)

Guardando trabajo temporal sin hacer commit

¿Qué es el Stashing?

Git stash te permite guardar temporalmente cambios sin hacer commit, útil cuando necesitas cambiar de rama rápidamente pero no quieres perder tu trabajo actual.

Casos de Uso Comunes:
  • Cambiar de rama con trabajo sin terminar
  • Actualizar desde el repositorio remoto
  • Probar una solución rápida
  • Guardar experimentos temporales
Comandos de Stash:
git stash

Guarda los cambios actuales en el stash

git stash save "Trabajo en progreso en login"

Guarda con un mensaje descriptivo

git stash list

Lista todos los stashes guardados

git stash pop

Restaura el último stash y lo elimina de la lista

git stash apply

Restaura el último stash pero lo mantiene en la lista

Flujo de Trabajo con Stash:

Trabajando

git stash

Cambiar rama

git stash pop

Resolución de Conflictos

Manejando cambios contradictorios en el código

¿Qué son los Conflictos?

Los conflictos ocurren cuando Git no puede fusionar automáticamente cambios porque múltiples personas han modificado las mismas líneas de código de manera diferente.

Cuándo Ocurren Conflictos:
  • Durante un merge entre ramas
  • Al hacer pull desde un repositorio remoto
  • Durante un rebase
  • Al aplicar un stash
Anatomía de un Conflicto:
<<<<<<< HEAD <title>Mi Proyecto Principal</title> ======= <title>Mi Proyecto con Nueva Característica</title> >>>>>>> feature-branch
Explicación de marcadores:
<<<<<<< HEAD: Inicio de tus cambios locales
=======: Separador entre versiones
>>>>>>> rama: Final de los cambios de la otra rama
Proceso de Resolución:
  1. Identificar archivos en conflicto:
    git status
  2. Editar archivos manualmente para resolver conflictos
  3. Marcar como resuelto:
    git add archivo-resuelto.html
  4. Completar el merge:
    git commit
Importante: Nunca hagas commit de un archivo que aún contenga marcadores de conflicto (<<<, ===, >>>)
Herramientas de Resolución:
  • Editor manual: VS Code, Sublime, etc.
  • Herramientas visuales: GitKraken, Sourcetree
  • Línea de comandos: vimdiff, meld

Repositorios Remotos

Colaborando con otros desarrolladores

¿Qué son los Repositorios Remotos?

Los repositorios remotos son versiones de tu proyecto alojadas en internet o en una red, permitiendo colaboración entre múltiples desarrolladores.

Conceptos Clave:
  • origin: Nombre por defecto del repositorio remoto principal
  • upstream: Repositorio original del cual hiciste fork
  • fetch: Descargar cambios sin fusionar
  • pull: Descargar y fusionar cambios
  • push: Subir cambios locales al remoto
Comandos Principales:
git remote -v

Muestra los repositorios remotos configurados

git pull origin main

Descarga y fusiona cambios de la rama main del remoto

git push origin feature/nueva-caracteristica

Sube tu rama de características al repositorio remoto

git fetch origin

Descarga cambios sin fusionar (más seguro)

Flujo de Colaboración:
1. Pull

Obtener últimos cambios

2. Develop

Trabajar en tu rama

3. Commit

Guardar cambios localmente

4. Push

Subir al repositorio remoto

Buena Práctica: Siempre haz git pull antes de comenzar a trabajar y antes de hacer git push

Rebase

Reescribiendo el historial para un proyecto más limpio

¿Qué es el Rebase?

Rebase es una alternativa al merge que "reescribe" el historial moviendo commits de una rama para que aparezcan como si hubieran sido creados desde el punto más reciente de otra rama.

Rebase vs Merge:
Con Merge:
main: A---B---C---F---G \ / feature: D---E-----

Preserva el historial real pero puede ser confuso

Con Rebase:
main: A---B---C---F---D'---E'

Historial lineal y más limpio

Comandos de Rebase:
git rebase main

Rebasa la rama actual sobre main

git rebase -i HEAD~3

Rebase interactivo para modificar los últimos 3 commits

Opciones del Rebase Interactivo:
Comando Descripción
pick Usar el commit tal como está
reword Usar el commit pero editar el mensaje
edit Usar el commit pero parar para editar
squash Combinar con el commit anterior
drop Eliminar el commit
¡Cuidado! Nunca hagas rebase de commits que ya han sido pusheados y son usados por otros desarrolladores.
Cuándo Usar Rebase:
  • ✅ En ramas de características locales
  • ✅ Para limpiar el historial antes de hacer merge
  • ✅ Para mantener un historial lineal
  • ❌ En ramas compartidas o públicas
  • ❌ Si no estás seguro de lo que haces

Workflows de Git

Metodologías para trabajar en equipo

Git Flow

Un modelo de branching que define roles específicos para diferentes tipos de ramas.

master: ●───●───●───●───● \ / release: ●───●───● \ / develop: ●───●───●───●───●───● \ / \ / feature: ●───● ●───● / \ hotfix: ●───●
Tipos de Ramas en Git Flow:
  • master/main: Código en producción
  • develop: Rama de desarrollo principal
  • feature/*: Nuevas características
  • release/*: Preparación de releases
  • hotfix/*: Correcciones urgentes

GitHub Flow

Un workflow más simple, ideal para proyectos con deploy continuo.

main

Siempre deployable

branch

Crear rama

commit

Hacer cambios

PR

Pull Request

review

Code Review

merge

Merge & Deploy

Mejores Prácticas:

✅ Buenas Prácticas:
  • Nombres descriptivos para ramas
  • Commits pequeños y frecuentes
  • Mensajes de commit claros
  • Testing antes de merge
  • Code review obligatorio
❌ Evitar:
  • Commits gigantes
  • Trabajar directamente en main
  • Forzar push en ramas compartidas
  • Merge sin testing
  • Ignorar conflictos
